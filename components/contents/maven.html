Nei progetti software tipicamente ci si imbatte in task come il download di dipendenze, inserimento di jar addizionali nei classpath, compilazione di codice sorgente in codice binario, lancio di test, packaging di codice per il deploy come JAR, WAR e ZIP, e il deploy di quest'ultimi in un application server o una repositoy.

Apache Maven automatizza questi task, minimizzando il rischio umano di fare errori quando si fa la build del software manualmente e separare il lavoro di compilazione e packaging dalla scrittura del codice in sé.

Maven utilizza il POM Project Object Model - scritto in linguaggio XML.

Le funzioni chiave di maven sono:

  • Semplificare il setup di un progetto usando le migliori pratiche, Maven prova a evitare la maggior parte delle configurazioni, usando i template di progetto nominati "archetypes".
  • Gestione delle dipendenze, include l'aggiornamento automatico, download e convalida, oltre a segnalare la chiusura delle dipendenze (nota come dipendenze transitive).
  • Isolamento tra dipendenze del progetto e plugins, con Maven le diependenze del progetto vengono recuperate dai repository, mentre le dipendenze di qualsiasi plug-in vengono recuperate dai repository dei plugin, determinando meno conflitti quando i plugin cominciano a scaricare dipendenze aggiuntive.
  • Sistema di repository centrale, le dipendenze del progetto possono essere caricate dal fle system locale o da repository pubblici come Maven Central.
La configurazione di un progetto Maven avviene con il POM, rappresentato dal file "pom.xml". Il POM descrive il progetto, gestisce le dipendenze e le configurazioni per la build del software. Il POM inoltre definisce la relazione tra i moduli di un progetto multi-modulo.

Per dichiarare una dipendenza ad una libreria esterna, bisogna indicare "groupid, artifactid e version". Questi tre componenti vengono anche chiamati coordinate, sono un identificatore per distinguere univocamente una libreria.
Le librerie esterne di un progetto vengono chiamate dipendenze. La funzionalità di gestione delle dipendenze di Maven garantisce il download automatico di tali librerie da un repository centrale, quindi non è necessario che venfano archiviate localmente.

Questa feature di Maven porte benefici in quanto si usa meno spazio riducendo in modo significativo il numero di download delle repository, velocizza il checking out di un progetto e fornisce una piattaforma efficace per lo scambio di artefatti all'interno dell'organizzazione.
Un repository di Maven viene utilizzato per contenere artefatti di build e dipendenze di vario tipo. Il repository locale predefinito si trova nella cartella ".m2/repository" nella directory home dell'utente.

Se nel repository locale è disponibile un artefatto o un plugin, Maven lo utilizza. In caso contrario viene scaricato da un repository centrale. Non tutte le librerie sono presenti nel repository centrale, per tali librerie sarà necessario fornire l'URL al repository alternativo all'interno del file "pom.xml".

Proprietà: Le proprietà aiutano il file "pom.xml" sulla leggibilità e mantenibilità. Un uso classico dove utilizzare le proprietà custom è per definire le versioni per le dipendenze del progetto. Le proprietà Maven sono segnaposto di valori e sono accessibili ovunque all'interno di un "pom.xml" utilizzando la notazione ${name}, dove name è la proprietà.

Build: La Sezione build del Maven POM, fornisce informazioni sull'obiettivo Maven predefinito, la directory del progetto compilato e il nome dell'applicazione.

Profili: Un profilo banalmente è un set di configurazioni. Usando i profili si può customizzare le varie build per i differenti environment come Produzione, Test e Development.

Ogni build di Maven segue un ciclo di vita specifico. Possiamo eseguire diversi obiettivi del ciclo di vita della build, inclusi quelli per compilare il codice del progetto, creare un pacchetto e installare l'artefatto prodotto.

Le fasi del ciclo di vita:

  • validate, controlla la correttezza del progetto.
  • compile, compila il codice sorgente fornito negli artefatti binari.
  • test, cesegue unit test.
  • package, packaging del codice compilato in file di archivi.
  • integration-test, esegue test addizionali, che richiedono il packaging.
  • verify, controlla se il package è valido.
  • install, installa il package nella repository locale di Maven.
  • deploy, deploy del pacchetto in un server remoto o nella repository.

Un progetto multi modulo è costituito con l'aggregazione di POM che gestiscono un gruppo di sottomoduli. In molti casi l'aggregatore è localizzato nella project's root directory. I sottomoduli sono dei regolari progetti Maven, e possono essere buildati separatamente attraverso l'aggregatore POM. Costruendo il progetto tramite l'aggregatore POM, ogni progetto che un tipo di packaging diverso risulterà in un archivio a parte.

I benefici di usare questo approccio è la riduzione di duplicati. In molti casi i progetti hanno bisogno di vari plugin di Maven per eseguire le varie operazioni durante il loro build lifecycle.

Submodules: I sottomoduli, o sottoprogetti, sono normali progetti Maven che ereditano dal POM padre. L'ereditarietà ci permette di condividere configurazioni e dipendenze con i sottomoduli.

BOM: Bill of Materials, un tipo speciale di POM usato per controllare le versioni delle dipendenze del progetto, provvede a centralizzare in un unico posto le versioni. Il BOM offre la flessibilità di aggiungere una dipendenza al nostro modulo senza preoccuparsi della versione da cui dovranno dipendere.
